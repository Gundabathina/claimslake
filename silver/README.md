# silver/

Local Silver layer output of the medallion architecture. The PySpark
Silver pipeline (see the `pyspark/` package) reads Bronze Parquet and
writes cleaned, validated, standardized data here as Parquet:

```
silver/
|-- members/                      one row per member (business key: member_id)
|-- providers/                    one row per (provider_id, effective_date) VERSION
|-- diagnoses/                    reference lookup (business key: diagnosis_code)
|-- claims/                       one row per claim_id, partitioned by service_year_month
|   |-- service_year_month=YYYY-MM/
|-- quarantine/                   records that failed validation, kept not dropped
    |-- members/
    |-- providers/
    |-- diagnoses/
    |-- claims/
```

## This directory is generated, not source-controlled

The Parquet files here are build artifacts, regenerated by running the
Silver pipeline locally, so they are excluded from git (see `.gitignore`).
Only this README is tracked, so the directory's purpose stays documented.

To populate it locally (Bronze must exist first):

```
pip install -r requirements.txt
python -m ingestion.src.ingestion_engine --all     # SOURCE  -> BRONZE
python -m pyspark.src.silver_pipeline --all         # BRONZE  -> SILVER
```

## What Silver does (and does not) do

Silver is where cleaning, standardization, type casting, deduplication,
referential-integrity checks, and quarantining happen. Unlike Bronze, it
does apply business rules. It still does NOT invent or "correct" values:
records that fail validation are routed to `silver/quarantine/` with a
`rejection_reason` and `validation_timestamp` rather than silently
dropped or patched.

Providers are deliberately NOT collapsed to one row per provider: rows
that represent real network-status/effective-date changes are preserved
as distinct versions so the future Gold layer can build an SCD Type 2
dimension. See `pyspark/README.md` and
`docs/interview_guide/03_pyspark_silver.md` for the full design.

All data is synthetic.
